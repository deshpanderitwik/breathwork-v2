Implementation details

How the breath app is wired up under the hood. Three platforms, one state machine, and a small but specific fix that makes the metronome stop wobbling.

Three platforms, one state machine

The web app, the macOS menu bar app, and the iOS app are three different shells around the same TypeScript module. The shared code lives at packages/core/ and is built into two artifacts: an ES module the web app imports directly, and an IIFE bundle the Apple apps load into a JavaScriptCore VM.

The choice is deliberate. The hard problem in this app is timing — making chimes land at the right millisecond, even after a pause, even after iOS interrupts the audio session. By keeping the schedule logic in one place, written in one language, I have one set of contract tests that govern all three apps. Drift between web and Swift becomes a test failure, not a heisenbug.

Audio synthesis stays native per platform — Web Audio for the browser, AVAudioEngine for Apple. The same TONE_DESIGN constants drive both, so the chime waveform is bit-identical across surfaces; the only thing that differs is the engine that renders it.

What the state machine does

At session start, the core compiles a fixed schedule of events:

{ kind: "inhale-count",  round: 1, beatIndex: 0, beatsInPhase: 4, atMs: 0 }
{ kind: "inhale-count",  round: 1, beatIndex: 1, beatsInPhase: 4, atMs: 1000 }
...
{ kind: "exhale-count",  round: 1, beatIndex: 0, beatsInPhase: 6, atMs: 4000 }
...
{ kind: "rest-start",    round: 1, durationSec: 30, fadeOutSec: 1.5, atMs: 90000 }
{ kind: "round-complete", round: 1, atMs: 120000 }
...
{ kind: "session-complete", atMs: 480000 }

The host (web app or Swift app) calls tick(nowMs) on a polling loop and gets back any events whose atMs has passed. Pause arithmetic is internal to the state machine — the host calls pause() / resume() at user gestures and the engine freezes effective time accordingly.

There are two cursors over the schedule:

  • Fired cursor — advanced by tick(). Drives the UI. Each event fires the moment it becomes audible. Used to swap the "Inhale" / "Exhale" label and update the round counter.
  • Audio cursor — advanced by tickAudio(nowMs, lookaheadMs). Returns events whose atMs falls inside a 150–250 ms lookahead window. Used to pre-schedule chimes against the audio engine's own clock — see "the chime stays in time" below.

The two cursors operate independently over the same schedule. The audio cursor is always at or ahead of the fired cursor; on pause, the host calls rewindAudioCursor() to pull it back so the cancelled chimes get re-scheduled on resume.

Three pieces in dialogue

The runtime architecture is three actors: the state machine, the poll loop, and the audio engine. The poll loop in the middle is the only active component — it pulls from the state machine and pushes to the audio engine.

The chime stays in time

The naive way to play a chime is "play it now, the moment the poll loop notices the event." That's how the app worked at first. The problem is that "now" depends on when the poll loop happens to fire — requestAnimationFrame on web runs every ~16 ms, Swift's Timer every ~100 ms. Each chime inherited a fresh random delay, audible as a stumble in the metronome.

The fix is to stop using the poll loop's clock for audio playback. Both Web Audio and AVAudioEngine accept absolute future timestamps in their own high-precision hardware clock.

  • Web: source.start(audioStart + sessionMs / 1000) against AudioContext.currentTime. Sample-accurate; the AudioContext clock never pauses.
  • Swift: player.scheduleBuffer(buf, at: AVAudioTime(sampleTime: anchor + …)) against the player's render sample-time. Sample-accurate too, but the anchor needs priming and re-anchoring; see iOS resilience below.

For this to work, the poll loop has to look slightly ahead — the audio engine can only schedule events it's been told about. So the state machine grew the second cursor (tickAudio) that returns events landing within a small lookahead window (150 ms on web, 250 ms on Swift to clear the slower Timer cadence). Each event the audio path returns is scheduled immediately at its precise atMs; the host holds a cancel handle so pause can flush anything queued past the freeze point.

The poll loop becomes a queue-filler, not a trigger. Its job narrows from "decide when each chime fires" to "tell the audio engine what's coming next." The hardware clock decides the audible moment.

iOS resilience

The Swift implementation has to handle several failure modes the web doesn't:

  • Anchor capture race. AVAudioPlayerNode.lastRenderTime is nil until the engine has rendered at least one buffer. prepare() schedules a 64-frame silent priming buffer to force a render. By the time beginSession() reads the anchor, it's reliably populated. A 50 ms headroom offset ensures sessionMs=0 events schedule comfortably in the future, not in the millisecond-old past.
  • Audio session interruptions. Phone calls, Siri, alarms post AVAudioSession.interruptionNotification. SessionController observes it: .began triggers a forced pause (cancel pending chimes, rewind audio cursor); .ended with .shouldResume triggers a forced resume after re-anchoring against the new effective time.
  • Engine reconfigs. Swapping AirPods or changing output devices fires AVAudioEngineConfigurationChangeNotification and silently stops the engine. The notification handler restarts the engine and re-anchors. The session keeps running; the user sees at most a brief gap.
  • Anchor invalidation. player.stop() resets the sample-time counter to 0, so any flush operation (pause, fadeOut at rest boundary) invalidates the anchor. The next scheduleAt() call lazily re-anchors using the current effective time, so the formula target = anchor + sessionMs * sampleRate / 1000 keeps producing correct sample-times even after a full pause cycle.

The end result: the chime stays in time across a phone call, an AirPods connect, a backgrounded screen lock, and ten minutes of rest-and-active rounds. The metronome locks to the audio clock, and the audio clock keeps running through anything that isn't a hardware reset.
